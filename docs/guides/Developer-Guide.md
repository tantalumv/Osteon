# Osteon Developer Guide: Programming at the Bone

Osteon is a **structured assembly language** designed for systems engineers who need the precision of assembly with the structural integrity of a modern language. It provides a 1:1 mapping to machine instructions while offering high-level abstractions for Data-Oriented Design (DOD), memory management, and error handling.

---

## 1. The Core Mental Model

In Osteon, **what you write is what executes.** 
- No implicit stack frames.
- No hidden prologues/epilogues.
- No automatic register saving.

However, Osteon provides **Transparent Sugar**. Constructs like `for` loops, `arena` allocation, and `struct` layouts are desugared at compile-time into raw instructions. You can audit this transformation at any time using:
```bash
osteon --dry-run your_code.ostn
```

---

## 2. Syntax Fundamentals

### Widths and Registers
Every instruction requires an explicit width annotation if it operates on registers or memory. This prevents "width mismatch" bugs.

```osteon
# mov(width) destination, source
mov(u64) rax, rbx          # 64-bit move
mov(u32) eax, imm(42)      # 32-bit move of immediate
add(u16) cx, dx            # 16-bit add
```

### Memory Access (`deref`)
Osteon uses a unified `deref` syntax for all memory addressing:
`deref(base, index, scale, displacement)`

```osteon
# Simple pointer access: [rax]
mov(u64) rbx, deref(rax)

# Complex SIB addressing: [rax + rcx*8 + 16]
mov(u64) rdx, deref(rax, rcx, 8, 16)
```

### Register Aliases (`let`)
Use `let` to give registers semantic names within a function. These disappear during compilation but make your logic readable.

```osteon
fn process_buffer {
    let buf_ptr = rdi
    let length  = rsi
    let cursor  = rax
    # ... use aliases instead of raw registers
}
```

## 3. Data-Oriented Design: Structs & SoA

Osteon treats structs as **memory layout definitions**, not objects.

### AoS (Array of Structs)
Standard layout. Osteon enforces explicit padding to ensure alignment is auditable.
```osteon
struct Entity {
    id:     u64,
    health: u32,
    _pad:   u32,  # Fatal error if omitted and alignment is broken
}
```

### SoA (Struct of Arrays)
The primary feature for SIMD and Cache-locality. Use `layout(soa)` to define a DOD block.
```osteon
layout(soa) struct Particle {
    x: f32,
    y: f32,
    z: f32,
}

const CAP = 1024
# Get the byte offset of the 'y' array within the SoA block
const Y_OFFSET = @soa_offset(Particle, y, CAP)
```

## 4. Built-in Arena Allocation

Osteon elevates the **Arena** to a language primitive. The compiler manages the bump-pointer arithmetic for you using a dedicated scratch register.

```osteon
fn init_system {
    # Initialize an arena using rdi as the base buffer
    arena mem = init(rdi, imm(4096))

    # Allocates 128 bytes, 16-byte aligned. Result in rax.
    alloc(mem, imm(128), imm(16)) 
    
    # Logic...
    
    reset(mem) # Moves bump pointer back to base
}
```
*Audit Note: Use `--dry-run` to see the exact `add/and/cmp/ja` instructions generated by `alloc`.*

## 5. Writing SIMD Kernels

Osteon provides direct access to `xmm`, `ymm`, and `zmm` registers. Combined with SoA layouts, writing high-performance kernels is straightforward.

```osteon
fn stream_add_f32 {
    let a_ptr = rdi
    let b_ptr = rsi
    let count = rdx # assume count is multiple of 8 (AVX)

    for rcx = imm(0), count, imm(8) {
        # Calculate offset for current iteration (count * 4 bytes per f32)
        imul(u64) r10, rcx, imm(4) 
        
        vmovups(f32) ymm0, deref(a_ptr, r10)
        vmovups(f32) ymm1, deref(b_ptr, r10)
        vaddps(f32)  ymm0, ymm0, ymm1
        vmovups(f32) deref(a_ptr, r10), ymm0
    }
    ret
}
```

## 6. Error Handling & Contracts

### The Result Convention
Osteon suggests a two-register convention for functions that can fail:
- `rax`: The return value.
- `rdx`: The error code (0 = Success).

### `expect` and `assert`
- `expect("msg")`: Checks if `rdx` is non-zero after a call. If it is, it traps (`ud2`).
- `assert(cmp, jcc, "msg")`: Runtime check that is **stripped** in `--release` builds.
- `static_assert(expr)`: Compile-time check.

```osteon
call calculate_physics
expect("Physics overflow") # Inline test(rdx) + jnz to trap
```

## 7. Interoperability & Linking

### Standalone Executable
To build a standalone `.exe` (Windows PE32+):
```bash
osteon --emit exe --out my_tool.exe main.ostn
```

### Static Linking (C/Odin/Zig)
Osteon emits standard COFF (`.obj`) files. You must respect the calling convention of your host language manually.

**1. Export from Osteon:**
```osteon
# math.ostn
fn fast_pow {
    # Microsoft x64: arg0 in rcx, arg1 in rdx
    # ... logic ...
    ret
}
```

**2. Compile to Object:**
```bash
osteon --emit obj math.ostn
```

**3. Import in Odin:**
```odin
foreign import "math.obj"
foreign math {
    fast_pow :: proc(base: f64, exp: f64) -> f64 ---
}
```

## 8. Compiler Workflow for Pros

1. **`osteon --check`**: Fast syntax and width-consistency check. Use this during active dev.
2. **`osteon --dry-run`**: Use this to verify that the compiler's desugaring of your `for` loops or `alloc` calls matches your performance expectations.
3. **`osteon --debug`**: Generates RADDBG info for use with RemedyBG or Raddbg.
4. **`osteon --release`**: Strips all `assert` and `breakpoint` instructions for production builds.

---

## 9. Best Practices

1.  **Always use `let`**: Don't track `r12` vs `r13` in your head. Name them.
2.  **Audit your loops**: For hot loops, check `--dry-run` to ensure the induction variable logic is tight.
3.  **Prefer SoA**: If you have more than 16 instances of a struct, use `layout(soa)` for cache efficiency.
4.  **Manual Stack Frames**: If you need a stack frame, you must `push rbp; mov rbp, rsp; sub rsp, N` yourself. Osteon will not do it for you.
